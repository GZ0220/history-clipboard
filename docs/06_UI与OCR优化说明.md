# UI 与 OCR 优化说明

## 本轮 UI 优化

- 主界面改为现代卡片型布局。
- 顶部工具栏保留搜索、排序、类型筛选和保存天数。
- 常用操作区保留暂停记录、显示识别结果、打开数据文件夹。
- 数据管理项折叠显示，包含导出备份、恢复备份、更换保存位置、状态诊断、清理未置顶。
- 设置入口单独放在顶部常用操作区；设置弹窗中显示“单张图片上限”和“最多记录”等选项。
- 批量操作条默认隐藏，选中记录后显示。
- OCR 识别结果默认不在卡片中展开，可通过“显示识别结果”切换为短预览。
- 点击窗口关闭按钮时弹出关闭方式选择，可最小化到系统托盘、完全关闭或取消。

## 流畅度优化

- 搜索输入使用短延迟刷新，避免每输入一个字都重绘列表。
- 多次连续刷新会合并到界面空闲时执行。
- 历史列表采用明确分页，每页只创建约 24 条卡片控件；滚动条只控制当前页，不再使用估算高度的上下大占位，避免滚动到顶部或底部时出现大片空白。
- 历史列表继续保留卡片复用，底部提供“上一页 / 下一页 / 第 x-y 条 / 共 n 条”，搜索、排序、筛选时不再一次性创建所有卡片。
- 历史卡片改为稳定控件结构，创建后保留按钮、标签、预览区和详情区引用；置顶、复制、备注、OCR 保存只更新对应字段。
- 置顶/取消置顶不再触发整表刷新，只更新当前卡片并按当前排序移动位置。
- 复制历史后只更新当前卡片的最近复制时间和必要位置，不再重建列表。
- 单条删除、批量删除、清理未置顶只移除对应卡片并更新数量。
- 图片缩略图加入内存缓存和后台预加载，当前页渲染后会预读前后记录段的缩略图，重复滚动时减少磁盘读取。
- 勾选、全选和取消选择只更新勾选状态与批量条，不再重建整张列表。
- “显示识别结果/隐藏识别结果”只更新当前屏幕中已渲染图片卡片的 OCR 预览区域，未渲染卡片滚动出现时再按当前设置显示。
- OCR 预览区域采用小批量分帧更新，图片记录较多时也尽量保持窗口响应。
- 剪贴板图片读取、图片哈希和大小判断放到后台线程，复制大图或截图时减少主窗口卡顿。
- 图片复制回剪贴板放到后台线程执行，并通过底部状态显示进度。
- 搜索文本和排序时间使用内存缓存，备注、OCR、正文变化时自动失效刷新。
- 批量删除、清理未置顶和最大记录数清理减少重复保存索引文件。
- 可通过环境变量 `CLIPBOARD_HISTORY_PERF=1` 启用开发用性能日志，观察刷新和局部更新耗时。
- 性能日志包含 `page_render`、`thumbnail_preload`，用于排查分页渲染和图片加载耗时。
- 增加图片大小上限，默认超过 5MB 的图片不记录。
- 增加最大历史数量，默认最多保留 500 条，超过后自动清理未置顶旧记录。
- 历史卡片固定为左侧预览、右侧信息和操作区。
- 日期独立显示为 `YYYY-MM-DD HH:MM:SS`，避免和类型文字挤在一起。
- 置顶记录使用更明显的视觉标签。
- 底部显示当前展示数量和总记录数量。
- 搜索时显示命中来源，例如正文、备注、OCR 或文件名。
- 双击历史卡片可直接复制回剪贴板。
- 最小化到托盘后主窗口隐藏，剪贴板记录继续运行；托盘菜单提供“打开历史粘贴板”和“完全退出”。
- 从托盘恢复时保留托盘图标常驻，只恢复主窗口并刷新当前虚拟列表窗口，减少黑边、白屏和二次加载感。

## 图片查看

- 图片卡片支持双击缩略图放大查看。
- 大图窗口支持：
  - 鼠标滚轮缩放。
  - 鼠标左键拖拽移动图片。
  - 适应窗口。
  - 复制图片。
  - 关闭窗口。
- 拖拽图片时只移动画布对象，不重新缩放原图；滚轮缩放会先快速预览，短暂停顿后再高清渲染。

## OCR 识别

当前版本使用 RapidOCR + ONNX Runtime 做离线图片文字识别。

选择离线 OCR 的原因：

- 图片不需要上传，更适合剪贴板隐私场景。
- 无网络时也能使用。
- 可以随便携版一起分发。

当前行为：

- 图片卡片点击“识别文字”后，后台线程执行识别，界面不会卡死。
- 识别结果会显示在弹窗中。
- 用户可以复制识别结果。
- 用户可以将识别结果保存到当前图片记录，保存后的 OCR 文本会参与搜索。
- 用户也可以将识别结果另存为文字卡片，这样主界面会直接显示一条新的文字历史。
- 便携发布包会包含 OCR 依赖和模型，因此包体比无 OCR 版本更大。
- 如果不需要 OCR，可以使用 `dist/历史粘贴板_轻量版/`，日常复制历史功能仍可使用。

## 隐私与管理

- 暂停记录只影响新复制内容，不影响已有历史。
- 清理未置顶记录会二次确认，置顶内容保留。
- 导出备份会在用户选择的目录下创建带时间戳的新文件夹，不覆盖旧备份。

候选方案记录：

- RapidOCR：当前采用，离线、体积相对可控，适合便携工具。
- PaddleOCR：能力强，但依赖更重，打包和模型管理复杂。
- EasyOCR：依赖 PyTorch，便携包体积和兼容性压力更大。
